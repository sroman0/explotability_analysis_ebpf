    \chapter{Introduction}

    \subsection{eBPF Background}

    Extended Berkeley Packet Filter (eBPF) is a Linux kernel technology that \textbf{allows small programs to be loaded dynamically and executed inside the kernel} without writing or loading traditional kernel modules. Modern eBPF is widely used for networking, tracing, observability and security monitoring because it provides programmable hooks while preserving near-native execution performance.

    An eBPF program is usually compiled from restricted C code into BPF bytecode. The bytecode is then submitted to the kernel through the \texttt{bpf()} system call. Before the program can be attached to a hook and executed, the \textbf{kernel validates it through the BPF Verifier}. Only after successful verification can the program be interpreted or JIT-compiled into native machine code.

    The eBPF execution model relies on a few core components:

    \begin{itemize}
        \item \textbf{Programs}: restricted bytecode executed at specific kernel hooks.
        \item \textbf{Maps}: shared data structures used to exchange data between userspace and kernel-space BPF programs.
        \item \textbf{Helpers}: kernel-provided functions exposed to BPF programs through a controlled API.
        \item \textbf{Program types}: execution contexts such as socket filters, traffic-control classifiers, tracing programs or LSM hooks.
    \end{itemize}

    This report focuses on vulnerabilities where the \textbf{verifier accepts programs whose runtime behavior violates the safety assumptions} checked during static analysis. This class of bugs is particularly relevant because eBPF code runs in kernel context after approval. Therefore, a verifier mistake can become a direct \textbf{kernel memory primitive} or a \textbf{denial-of-service condition}.

    \subsection{The BPF Verifier}

    The BPF Verifier is the \textbf{security boundary between user-controlled BPF bytecode and privileged kernel execution}. Its purpose is to reject programs that may access invalid memory, use unsafe pointer arithmetic, leak kernel pointers, execute unbounded loops or call helpers with invalid arguments.

    At a high level, the verifier performs major checks like:

    \begin{enumerate}
        \item \textbf{Control-flow validation}: the program must terminate and must not contain invalid jumps or unreachable unsafe paths.
        \item \textbf{Register and pointer tracking}: each register is associated with a verifier state, represented by \texttt{struct bpf\_reg\_state}. This state records whether the register is uninitialized, a scalar value, or a pointer type such as \texttt{PTR\_TO\_MAP\_VALUE}, \texttt{PTR\_TO\_PACKET}, or \texttt{PTR\_TO\_STACK}.
        \item \textbf{Bounds tracking}: scalar ranges, pointer offsets and accessible memory windows are tracked to prevent out-of-bounds accesses.
    \end{enumerate}

    The verifier does not simply inspect individual instructions. It symbolically simulates possible execution paths and maintains an abstract state for registers, stack slots and memory references. If the abstract model is correct, unsafe programs are rejected before execution. If the model is wrong, the runtime program may operate on memory that the verifier incorrectly classified as safe.

    The vulnerabilities analyzed in this report exploit exactly this gap between abstract verification and concrete execution:

    \begin{itemize}
        \item \textbf{CVE-2023-39191}: stale \texttt{dynptr} stack tracking produces an OOB read/write primitive.
        \item \textbf{CVE-2024-58100}: missing EXT-program compatibility check on \texttt{changes\_pkt\_data} leaves a stale packet pointer alive.
        \item \textbf{CVE-2024-42072}: incorrect \texttt{may\_goto} handling turns a bounded loop into an infinite kernel loop.
        \item \textbf{CVE-2024-45020}: false pruning in \texttt{stacksafe()} produces a verifier-side OOB read and an information leak.
    \end{itemize}

    \subsection{Testing Environment}

    All experiments were executed inside a \textbf{controlled Buildroot/QEMU environment}. Buildroot was used to generate \textbf{minimal Linux images} with specific vulnerable kernel versions and only the userspace components required to compile, load and trigger the BPF programs. QEMU was used to execute the generated images in an isolated virtual machine.

    This setup provides three important advantages. First, each CVE can be tested against the exact kernel version required by the vulnerability. Second, the environment is \textbf{deterministic enough to reproduce heap layout} and memory placement assumptions. Third, crashes, soft lockups or kernel panics remain \textbf{isolated inside the VM}.

    The general workflow used for each PoC was:

    \begin{lstlisting}[caption={Generic Buildroot/QEMU testing workflow.}]
    Compile the PoC or exploit statically.
    Copy the binary into the Buildroot overlay.
    Build the selected vulnerable kernel image.
    Boot the image in QEMU.
    Execute the PoC inside the VM.
    Compare the result with the expected vulnerable behavior.
    \end{lstlisting}

    PoC binaries were \textbf{compiled statically to avoid dependency issues} inside the minimal Buildroot filesystem:

    \begin{lstlisting}[language=bash, caption={Static compilation of a PoC.}]
    buildroot/output/host/bin/x86_64-buildroot-linux-gnu-gcc -o CVEs/CVE-202x-xxxx/exploit_overlay/root/poc CVEs/CVE-202x-xxxx/src/poc.c
    \end{lstlisting}

    The resulting executable was copied into the exploit overlay before building the image, so that it became available inside the VM at boot time.

    \subsection{VM Configuration}

    The VM configuration was \textbf{adapted to the requirements of each CVE}. Some vulnerabilities only require a minimal amount of memory and a specific vulnerable kernel version. Others, especially CVE-2023-39191 and CVE-2024-58100, require additional \textbf{control over memory layout, kernel symbols, BPF permissions} and userspace capabilities.

    The common test environment used the following baseline properties:

    \begin{table}[H]
    \centering
    \begin{tabularx}{\textwidth}{|l|X|}
    \hline
    \textbf{Component} & \textbf{Configuration} \\
    \hline
    Virtualization & QEMU virtual machine generated through Buildroot \\
    \hline
    Userspace & Minimal Buildroot root filesystem \\
    \hline
    BPF availability & \texttt{CONFIG\_BPF} and \texttt{CONFIG\_BPF\_SYSCALL} enabled \\
    \hline
    Execution mode & Serial console, root login available for controlled testing \\
    \hline
    PoC/exploit format & C binaries built with Buildroot's cross-toolchain and deployed via overlay \\
    \hline
    Result validation & Verifier acceptance/rejection, kernel logs, map leaks, crash, soft lockup, or privilege escalation \\
    \hline
    \end{tabularx}
    \caption{Baseline testing environment.}
    \end{table}

    For CVE-2023-39191, the VM required a larger memory configuration. The exploit uses an OOB window that reaches approximately the 8.2 GiB physical memory region. For this reason, the VM was launched with 9 GiB of RAM:

      \begin{lstlisting}[language=bash, caption={QEMU memory configuration used for CVE-2023-39191.}]
      ./start-qemu.sh --serial-only -- -m 9216
      \end{lstlisting}

    In this configuration, \textbf{address randomization was disabled} to keep the \textbf{memory layout stable} during exploit development and testing. In particular, KASLR and \texttt{RANDOMIZE\_MEMORY} were disabled. This does not remove the vulnerability itself, but makes the exploit path reproducible enough to study the primitive, calibrate memory pressure and verify the credential overwrite.

    \subsection{Result Interpretation}

    The outcome of each experiment was interpreted according to the vulnerability class.

    For verifier bypass and memory-safety bugs, a kernel was considered vulnerable when a \textbf{program was accepted even though it should have been rejected}, or when a runtime primitive such as OOB read/write, stale pointer access or information disclosure was observed.

    For availability bugs, such as CVE-2024-42072, success was defined by the ability to \textbf{trigger the expected kernel soft lockup} after loading and executing the crafted BPF program.

    For privilege-escalation bugs, success required demonstrating a transition from a non-root execution context to root privileges. In CVE-2023-39191, this was achieved by locating and overwriting a live \texttt{struct cred} object and creating a SUID-root shell payload.

    The following table summarizes the main experimental requirements for the analyzed CVEs:

    \begin{table}[H]
    \centering
    \begin{tabularx}{\textwidth}{|l|X|X|}
    \hline
    \textbf{CVE} & \textbf{Main environment requirement} & \textbf{Observed result} \\
    \hline
    CVE-2023-39191 & 9 GiB RAM, stable memory layout, unprivileged BPF enabled & Full LPE through OOB read/write and \texttt{struct cred} overwrite \\
    \hline
    CVE-2024-58100 & Boot-time \texttt{S99exploit} setup, BPF/network capabilities, readable kernel symbols & Stable UAF R/W primitive; final unprivileged race not fully reliable \\
    \hline
    CVE-2024-42072 & Vulnerable 6.9 kernel with \texttt{may\_goto} support & Kernel soft lockup / DoS \\
    \hline
    CVE-2024-45020 & BPF-capable execution context, verifier pruning trigger & Information leak / KASLR bypass; unprivileged exploitation blocked in tests \\
    \hline
    \end{tabularx}
    \caption{CVE-specific testing requirements and outcomes.}
    \end{table}
